Contexte

Vous allez déployer une application web Mini Todo sur votre cluster k3d (déjà installé en TP précédent). L'application est composée de deux services :

  • un backend Node.js — API REST, stockage en mémoire
  • un frontend statique (nginx + HTML/JS) qui consomme l'API
Vous n'avez PAS à développer ces applications. Les images Docker sont déjà publiées sur Docker Hub :
  • stephanparichon/epsi-k8s-bff:1.0
  • stephanparichon/epsi-k8s-front:1.0
  • stephanparichon/epsi-k8s-front:2.0 (pour le rolling update)

Votre travail consiste uniquement à écrire les manifests Kubernetes qui les déploient.

Perte de données — point important

Le backend stocke les todos en mémoire vive du pod. À chaque redémarrage du pod backend (rolling update, scaling, crash, kubectl delete pod), toutes les todos sont effacées. C'est volontaire : ce TD ne couvre pas la persistance (PVC, StatefulSet). Pensez-y avant de tester vos rollouts !

Étape 1 — Secret 3 PTS

Objectif

Stocker en cluster un token administrateur consommé par le backend, sans l'écrire en clair dans le manifest du Deployment ni dans le repo Git.

Méthode — création impérative

Comme vu au module 7.2 du TP-k3d, on crée les Secrets en mode impératif (en ligne de commande) — pour ne jamais committer de credentials en clair dans un fichier YAML versionné sur Git.

Pour ce TD, il n'y a donc pas de fichier manifests/01-secret.yaml à écrire. La commande exacte que vous avez exécutée figurera dans le README de votre rendu (étape 5).

Cahier des charges

Créez un Secret générique (mode impératif) qui contient :

  • Nom du Secret : backend-secret
  • Une clé : admin-token
  • Valeur exacte : s3cr3t-token-td

Comment vérifier votre travail

  • Listez les Secrets de votre namespace courant — backend-secret doit apparaître
  • Son type doit être Opaque et il doit contenir une seule donnée (colonne DATA = 1)
  • Si vous voulez aller plus loin : récupérez le contenu du Secret et décodez la valeur base64 — vous devez retomber sur la valeur exacte du cahier des charges

Livrable : la commande exacte que vous avez exécutée doit apparaître dans le README de rendu (cf. section 9).

Étape 2 — ConfigMap 3 PTS

Objectif

Externaliser la configuration du frontend (où joindre le backend ?) dans un objet K8s, plutôt qu'en dur dans l'image.

Cahier des charges

  • Fichier : manifests/02-configmap.yaml
  • Nom : frontend-config
  • Deux clés dans data :
    • – BACKEND_HOST : nom du Service backend dans le cluster
    • – BACKEND_PORT : port d'écoute du Service backend

Rappel — DNS interne du cluster

Dans le même namespace, le nom court du Service suffit (résolu par CoreDNS). Hors namespace : <svc>.<ns>.svc.cluster.local.

Référez-vous à 05-service-back.yaml pour trouver les bonnes valeurs.

Comment vérifier votre travail

  • Appliquez votre manifest dans le cluster
  • Listez les ConfigMaps — frontend-config doit apparaître avec DATA = 2 (les deux clés)
  • Inspectez son contenu en YAML pour confirmer que BACKEND_HOST et BACKEND_PORT sont présentes avec les bonnes valeurs

Étape 3 — Deployment backend 4 PTS

Objectif

Déployer le backend Node.js avec auto-healing, health check et injection sécurisée du token via Secret.

Cahier des charges

Ouvrez le squelette manifests/03-deployment-back.yaml et remplissez les # TODO.

  • replicas: 1
  • Image : stephanparichon/epsi-k8s-bff:1.0
  • Port conteneur : 3000
  • Labels app=backend cohérents entre selector et template
  • Variable d'env ADMIN_TOKEN injectée depuis le Secret de l'étape 1 (clé admin-token)
  • readinessProbe HTTP GET /health sur le port 3000 (initialDelaySeconds 3, periodSeconds 5)
Le backend refuse de démarrer si ADMIN_TOKEN n'est pas défini. Si vous voyez le pod en CrashLoopBackOff, lisez les logs du pod backend pour confirmer la cause.

Comment vérifier votre travail

  • Appliquez votre manifest. Attention : le Secret de l'étape 1 doit déjà exister, sinon le pod crashloopera
  • Surveillez le déploiement du rollout : il doit aboutir sans erreur
  • Listez les pods filtrés par le label app=backend — vous devez voir 1 pod en Running, READY 1/1
  • Si le pod est en CrashLoopBackOff : lisez ses logs pour identifier la cause (Secret mal référencé ? clé incorrecte ?)

Étape 4 — Deployment frontend 4 PTS

Objectif

Déployer le frontend en haute disponibilité (2 réplicas), consommer la ConfigMap, et préparer un rolling update sans interruption.

Cahier des charges

Écrivez from scratch manifests/04-deployment-front.yaml en vous inspirant du back.

  • Nom : frontend
  • replicas: 2
  • Image : stephanparichon/epsi-k8s-front:1.0
  • Port conteneur : 80
  • Variables BACKEND_HOST et BACKEND_PORT injectées depuis la ConfigMap frontend-config via envFrom: configMapRef
  • readinessProbe HTTP GET /health sur le port 80 (initialDelaySeconds 2, periodSeconds 5)
  • Stratégie de rolling update : maxUnavailable: 0, maxSurge: 1

Rappel — envFrom vs env

envFrom + configMapRef injecte toutes les clés de la ConfigMap comme variables d'env, d'un coup. env + valueFrom + configMapKeyRef serait à utiliser pour une seule clé. Ici on veut les deux clés → envFrom est plus concis.

Attention : avant d'appliquer ce Deployment, le Service backend (05-service-back.yaml, fourni) doit déjà exister dans le cluster. Sinon nginx crashe au démarrage avec host not found in upstream "backend" (nginx résout les DNS au démarrage, pas à chaque requête). C'est la symétrie du piège de l'étape 3 avec le Secret. Si vous découvrez le problème après coup, un kubectl rollout restart deployment/frontend force la résolution une fois le Service appliqué.

Comment vérifier votre travail

  • Appliquez votre manifest et surveillez le rollout jusqu'à sa fin
  • Listez les pods filtrés par le label app=frontend — vous devez voir 2 pods en Running, chacun avec READY 1/1
  • Inspectez un pod pour confirmer que les variables d'environnement BACKEND_HOST et BACKEND_PORT y sont bien injectées (depuis la ConfigMap)
  • Inspectez la spec du Deployment pour vérifier que la stratégie de rolling update est bien configurée comme demandé

Appliquer, tester, mettre à jour

Étape 5 — Tout appliquer et tester 2 PTS

  1. Créez le Secret de l'étape 1 (mode impératif).
  2. Appliquez les manifests YAML du dossier manifests/. L'ordre d'application a son importance : un pod qui démarre avant que ses dépendances (Secret, Service à résoudre via DNS…) existent peut crasher. À vous de trouver le bon ordre.
  3. Vérifiez que tout est en place : pods Running et Ready, rollouts terminés, ressources attendues présentes dans le cluster.

Ouvrez http://localhost:8080. L'application doit charger. Ajoutez et supprimez quelques todos.

La ligne Backend pod : backend-xxxx en haut de l'app vous indique quel pod backend a répondu.

Étape 6 — Rolling update à observer 2 PTS

Méthode déclarative — on modifie le manifest YAML (la source de vérité versionnée dans Git), puis on applique. C'est l'approche GitOps : le repo reflète toujours l'état désiré du cluster.

Avant de lancer la mise à jour, ouvrez dans un terminal séparé une commande qui surveille en temps réel l'évolution des pods, pour observer le rolling update se dérouler.

1. Éditez le fichier manifests/04-deployment-front.yaml et changez la version de l'image du conteneur frontend de epsi-k8s-front:1.0 vers epsi-k8s-front:2.0.

2. Demandez à Kubernetes de réconcilier l'état du cluster avec votre manifest modifié, puis suivez le rollout jusqu'à sa fin.

3. Rafraîchissez le navigateur → vous devez voir le thème violet/orange et le badge "v2.0 — NEW".

Capture obligatoire : deux screenshots — avant (v1 fond bleu) et après (v2 fond violet) — à joindre au rendu Git. Et n'oubliez pas de committer la nouvelle version du YAML dans votre repo.

Étape 7 — Tester le Secret avec curl 2 PTS

Le backend expose une route administrateur protégée par un en-tête HTTP qui doit contenir le token de l'étape 1 :

  • Méthode : POST
  • Route : /api/admin/clear (exposée via le frontend à http://localhost:8080)
  • En-tête attendu : X-Admin-Token

Avec curl, démontrez les deux cas :

  1. Sans en-tête (ou avec un token erroné) → la route doit refuser la requête (401).
  2. Avec l'en-tête contenant la bonne valeur → la route doit répondre OK (200) et vider les todos.

Rafraîchissez le navigateur : les todos doivent avoir disparu.
