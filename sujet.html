Contexte

Vous allez déployer une application web Mini Todo. L'application est composée de deux services :

  • un backend Node.js — API REST, stockage en mémoire
  • un frontend statique (nginx + HTML/JS) qui consomme l'API
Vous n'avez PAS à développer ces applications. Les images Docker sont déjà publiées sur Docker Hub :
  • stephanparichon/epsi-k8s-bff:1.0
  • stephanparichon/epsi-k8s-front:1.0
  • stephanparichon/epsi-k8s-front:2.0 (pour le rolling update)

Votre travail consiste uniquement à écrire les manifests Kubernetes qui les déploient — et à savoir diagnostiquer quand ça ne marche pas.

Perte de données — point important

Le backend stocke les todos en mémoire vive du pod. À chaque redémarrage du pod backend (rolling update, scaling, crash, kubectl delete pod), toutes les todos sont effacées.

        └─────────┬──────────┘
                  │ proxy_pass /api/

   └──────────────┬───────────────────┘
                  │

Ce schéma est votre cahier des charges d'architecture. Les noms (backend, frontend, frontend-config…), les ports et les types de Service qui y figurent doivent être cohérents dans tous vos manifests.

2

Contrat des images

Vous déployez des images que vous n'avez pas écrites. Voici tout ce que vous devez savoir sur leur comportement pour les câbler correctement. C'est votre seule source de vérité : aucun squelette YAML ne vous est fourni.

Backend

stephanparichon/epsi-k8s-bff:1.0

  • Écoute sur le port 3000
  • Health check : GET /health → 200 quand il est prêt
  • Exige la variable d'env ADMIN_TOKEN au démarrage → refuse de démarrer si absente (CrashLoopBackOff)
  • Route protégée : POST /api/admin/clear, en-tête X-Admin-Token requis (vide les todos)
  • Stockage en mémoire (todos perdus à chaque redémarrage)

Frontend

stephanparichon/epsi-k8s-front:1.0 / :2.0

  • Écoute sur le port 80
  • Health check : GET /health
  • Exige deux variables d'env : BACKEND_HOST et BACKEND_PORT
  • Proxifie /api/ vers le backend
  • nginx résout le DNS du backend AU DÉMARRAGE, pas à chaque requête. Retenez ce fait : il explique certains échecs.

Le dossier manifests/

Le repo fourni ne contient presque rien : à vous d'écrire les manifests. Voici l'état de départ et ce que vous devez produire.

Fichier État de départ
service-back.yaml Fourni comme exemple de référence
Le fichier service-back.yaml est votre modèle : il montre à quoi ressemble un Service bien formé (labels, selector, ports). Servez-vous-en pour écrire le Service frontend et pour vérifier la cohérence de vos labels.

À produire

Étape 1 — Secret 2 PTS

Stocker dans le cluster un token administrateur consommé par le backend.

Contraintes :

  • Nom du Secret : backend-secret
  • Une clé : admin-token
  • Valeur exacte : s3cr3t-token-td

À vous de déterminer comment créer le Secret. La commande exacte devra figurer dans votre README.

Étape 2 — ConfigMap 2 PTS

Externaliser la configuration du frontend (comment joindre le backend dans le cluster ?) dans un objet Kubernetes, plutôt qu'en dur dans l'image.

Contraintes :

  • Fichier : manifests/configmap.yaml, nom : frontend-config
  • Deux clés : BACKEND_HOST et BACKEND_PORT
  • Les valeurs doivent permettre au frontend de joindre le backend dans le cluster. À vous de les déduire du Service backend (fichier service-back.yaml) et du DNS interne.

Rappel DNS. Dans le même namespace, le nom court du Service est résolu par CoreDNS. Hors namespace : <svc>.<ns>.svc.cluster.local. Quel nom et quel port faut-il donc mettre ?

Étape 3 — Deployment backend 3 PTS

Déployer le backend en deployment-back.yaml, from scratch, de façon auto-réparée, avec une injection sécurisée du token et un routage du trafic seulement quand le pod est prêt.

Votre Deployment doit garantir que le backend :

  • tourne en 1 exemplaire, redémarré automatiquement s'il meurt ;
  • utilise l'image et le port définis dans le contrat (section 2) ;
  • porte des labels cohérents entre selector et template, et compatibles avec le Service backend fourni ;
  • reçoit son ADMIN_TOKEN
  • ne reçoit du trafic que lorsqu'il répond réellement sur son endpoint de santé.

Auto-vérification

  • 1 pod backend en Running, READY 1/1

Étape 4 — Deployment frontend 3 PTS

Déployer le frontend en deployment-front.yaml, from scratch, en haute disponibilité, consommant la ConfigMap.

Votre Deployment doit garantir que le frontend :

  • s'appelle frontend, tourne en 2 exemplaires ;
  • utilise l'image stephanparichon/epsi-k8s-front:1.0 et le port du contrat ;
  • reçoit BACKEND_HOST et BACKEND_PORT depuis la ConfigMap frontend-config.
  • n'accepte du trafic qu'une fois prêt (readiness sur son endpoint de santé, initialDelay ~2 s) ;
  • se met à jour sans jamais rendre un pod indisponible : à vous de traduire ça en stratégie de rolling update (pensez maxUnavailable / maxSurge).
Pour rappel : nginx résout le DNS du backend au démarrage.

Étape 5 — Services 2 PTS

Exposer les deux Deployments avec les bons types de Service, en cohérence avec l'architecture (section 2).

  • Service backend : fourni en service-back.yaml.
  • Service frontend : il doit exposer hors du cluster (via le loadbalancer k3d, accès http://localhost:8080).
5

Appliquer, tester, mettre à jour

Étape 6 — Tout appliquer et tester 1 PT

  • Ouvrez http://localhost:8080. L'application doit charger. Ajoutez et supprimez quelques todos. La ligne Backend pod : backend-xxxx en haut indique quel pod backend a répondu.
  • Prendre des captures d'écrans avec des TODO + nom des pods backend qui sont restitué.

Étape 7 — Rolling update v1 → v2 1 PT

Dans un terminal séparé, lancez une commande qui surveille les pods en temps réel, puis :

  1. Éditez deployment-front.yaml : image front:1.0 → front:2.0.
  2. Réconciliez le cluster avec votre manifest, puis suivez le rollout jusqu'à la fin.
  3. Rafraîchissez le navigateur → thème violet/orange et badge « v2.0 — NEW ».
Captures obligatoires : avant (v1 bleu) et après (v2 violet). Committez la nouvelle version du YAML.

Étape 8 — Déploiement raté & rollback 2 PTS

Constater qu'une mauvaise version ne casse pas la production grâce à votre stratégie de rolling update, puis revenir en arrière proprement.

Gardez votre surveillance des pods active, puis :

  1. Éditez à nouveau deployment-front.yaml et pointez vers une version inexistante.
  2. Observez : le nouveau pod reste bloqué (ImagePullBackOff), il ne devient jamais Ready. Que fait le rollout ? Que voit l'utilisateur sur http://localhost:8080 pendant ce temps ?
  3. Revenez à la version qui marche, soit via un rollback natif de Kubernetes, soit en réappliquant le bon YAML. Justifiez votre choix dans le README.
Captures obligatoires : du déploiement en erreur et de son retour.

Étape 9 — Tester le Secret avec curl 1 PT

Le backend expose une route admin protégée (cf. contrat, section 2) : POST /api/admin/clear, exposée via le frontend à http://localhost:8080, en-tête X-Admin-Token.

Avec curl, démontrez les deux cas :

  1. Sans en-tête (ou token erroné) → la route refuse (401).
  2. Avec le bon token → la route répond OK (200) et vide les todos.

Rafraîchissez le navigateur : les todos doivent avoir disparu.

Livrable

Dépôt Git public (GitHub ou GitLab)

Le dépôt doit contenir l'arborescence suivante :

votre-repo/
├── manifests/
│   ├── configmap.yaml            (écrit par vous)
│   ├── deployment-back.yaml      (écrit par vous)
│   ├── deployment-front.yaml     (écrit par vous, remis en v2.0)
│   ├── service-back.yaml         (fourni — réutilisé tel quel)
│   └── service-front.yaml        (écrit par vous)
├── screenshots/
│   ├── ...
└── README.md

Le fichier cassé 98-backend-broken.yaml n'est pas à committer corrigé : son analyse va dans le README.
