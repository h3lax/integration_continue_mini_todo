Pré-requis — TD noté Kubernetes

À faire AVANT le jour du TD · Compte ~30 min selon votre connexion

0. Avant tout 1. Docker 2. k3d 3. kubectl 4. Cluster TD 5. Pré-tirage images 6. Smoke test 7. Si ça coince

Checklist du jour J — vous devez pouvoir cocher tout ça

docker version retourne un client et un serveur (daemon UP)
k3d version retourne une version (5.x ou supérieure)
kubectl version --client retourne une version

1

Docker

Objectif

Avoir un daemon Docker fonctionnel. k3d fait tourner k3s dans des conteneurs Docker : pas de Docker = pas de cluster.

1.1 — Vérifier d'abord si Docker est déjà là

docker version

Vous devez voir deux blocs (Client + Server). Si seul le Client apparaît avec une erreur "Cannot connect to the Docker daemon", démarrez Docker Desktop puis retestez.

# Test plus parlant : lancer un conteneur réel
docker run --rm hello-world
# Doit afficher "Hello from Docker!"

1.2 — Installer Docker Desktop si absent

Windows 10/11

Pré-requis : WSL 2 activé (PowerShell admin : wsl --install)

docs.docker.com — Windows installer
macOS

Choisir la bonne archi : Apple Silicon (M1/M2/M3) ou Intel

docs.docker.com — macOS installer
Linux

Docker Desktop OU Docker Engine selon distrib

docs.docker.com — Linux installer
Après installation Docker Desktop sous Windows : redémarrez votre PC. Sans redémarrage, WSL 2 n'est pas pleinement activé et Docker démarre en boucle.

Validation

docker version
docker run --rm hello-world

Les deux doivent fonctionner sans erreur.

2

k3d

Objectif

Installer la CLI k3d qui crée et gère vos clusters Kubernetes locaux à partir d'images Docker.

2.1 — Vérifier si k3d est déjà installé

k3d version
# Attendu : k3d version v5.x.x

Si la commande est introuvable, installez via l'une des méthodes ci-dessous selon votre OS.

2.2 — Installation par OS

Windows (PowerShell)
# Option A — winget (Windows 10+)
winget install k3d-io.k3d
# Option B — Chocolatey si vous l'utilisez déjà
choco install k3d
# Option C — Scoop
scoop install k3d

Après installation, fermez et rouvrez votre terminal pour que le PATH soit rechargé.

macOS
brew install k3d
Linux
curl -s https://raw.githubusercontent.com/k3d-io/k3d/main/install.sh | bash

Validation

k3d version
# Doit afficher une version (5.x ou supérieure)

4

Création du cluster du TD

Objectif

Créer un cluster td-k8s avec le mapping de port 8080:80 qui rendra l'application accessible sur http://localhost:8080.

Le flag --port est obligatoire

Si vous oubliez --port "8080:80@loadbalancer" à la création, votre Service LoadBalancer sera injoignable depuis le navigateur, même avec une External IP affichée. Il n'y a pas de moyen simple d'ajouter ce mapping après coup : il faut détruire et recréer le cluster.

4.1 — Nettoyer un éventuel ancien cluster

# Lister les clusters existants
k3d cluster list
# Supprimer un cluster nommé td-k8s s'il existe
k3d cluster delete td-k8s

4.2 — Créer le cluster du TD

k3d cluster create td-k8s \
  --servers 1 --agents 2 \
  --port "8080:80@loadbalancer" \
  --k3s-arg "--disable=traefik@server:0"
Décryptage des flags
  • --servers 1 : 1 nœud control-plane
  • --agents 2 : 2 workers
  • --port : expose le port 80 du LB k3d sur le 8080 de votre PC
  • --k3s-arg "--disable=traefik..." : désactive l'Ingress par défaut (inutile pour ce TD)
Sur PowerShell Windows

Le \ de fin de ligne ne fonctionne pas. Mettez la commande sur une seule ligne, ou utilisez le backtick ` comme caractère de continuation PowerShell.

Validation

kubectl get nodes
# Attendu : 3 noeuds en statut Ready (1 control-plane + 2 sans rôle)
docker ps --filter "name=k3d-td-k8s"
# Attendu : 4 conteneurs (server + 2 agents + loadbalancer)
5

Pré-tirage des images Docker Hub

Pourquoi le faire à la maison

Le jour du TD, votre cluster k3d tirera ces images depuis Docker Hub la première fois que vous appliquerez les Deployments. Avec le wifi de la salle partagé entre 30 étudiants, ça peut prendre 3 à 10 minutes. En tirant à la maison, vous transférez les images dans le cluster en quelques secondes.

5.1 — Tirer les images sur votre machine

docker pull stephanparichon/epsi-k8s-bff:1.0
docker pull stephanparichon/epsi-k8s-front:1.0
docker pull stephanparichon/epsi-k8s-front:2.0

Compte ~200 Mo au total. Vérifiez ensuite :

docker images | grep epsi-k8s
# Attendu : 3 lignes (bff:1.0, front:1.0, front:2.0)

5.2 — Importer les images dans le cluster k3d

Important : k3d a son propre registre interne. Tirer une image sur votre machine ne la rend pas disponible pour les pods. Il faut soit la laisser être tirée par le cluster au déploiement (long), soit l'importer manuellement (instantané) :

k3d image import \
  stephanparichon/epsi-k8s-bff:1.0 \
  stephanparichon/epsi-k8s-front:1.0 \
  stephanparichon/epsi-k8s-front:2.0 \
  --cluster td-k8s

Sur PowerShell Windows : mettre la commande sur une seule ligne ou utiliser le backtick ` à la place du \.

Validation

# Vérifier la présence des images dans un node k3d
docker exec k3d-td-k8s-server-0 crictl images | grep epsi-k8s
# Attendu : 3 lignes correspondant aux 3 images importées
6

Smoke test final

Objectif

Déployer un nginx jetable pour valider de bout en bout que toute la chaîne fonctionne : Docker → k3d → cluster → pod → service → port mapping → navigateur.

6.1 — Déployer un nginx temporaire

# Créer un pod nginx
kubectl run smoke-nginx --image=nginx:1.27-alpine --port=80 --labels="app=smoke"
# Exposer le pod en LoadBalancer (utilise le port mapping 8080:80)
kubectl expose pod smoke-nginx --type=LoadBalancer --port=80

6.2 — Vérifier

kubectl get pod,svc
# Attendu : pod smoke-nginx en Running 1/1, service en LoadBalancer

Ouvrez ensuite http://localhost:8080 dans votre navigateur — la page "Welcome to nginx!" doit s'afficher.

6.3 — Nettoyer le smoke test

kubectl delete pod smoke-nginx
kubectl delete svc smoke-nginx

Le cluster reste prêt pour le jour du TD.

Si la page nginx s'est affichée sur localhost:8080

Votre environnement est 100% prêt pour le TD. Ne touchez plus à rien — gardez le cluster en place ou supprimez-le et recréez-le le jour J avec la même commande.

7

Si ça coince

Docker Desktop ne démarre pas, tourne en boucle

1. Vérifier WSL 2 (Windows) : wsl --status doit afficher "Version par défaut : 2"

2. Si WSL 2 inactif : ouvrir PowerShell en admin et lancer wsl --install, puis redémarrer le PC

3. Vérifier que la virtualisation est activée dans le BIOS/UEFI (VT-x ou AMD-V)

4. Vérifier que Hyper-V est activé (Fonctionnalités Windows)

k3d cluster create échoue : "port already allocated" sur 8080

Une autre application utilise déjà le port 8080. Identifiez-la :

netstat -ano | findstr :8080
# sous Linux/macOS : lsof -i :8080

Solution : fermer l'application coupable, OU créer le cluster sur un autre port local :

--port "9090:80@loadbalancer"
# L'app sera alors sur http://localhost:9090
k3d cluster create échoue avec "no space left on device"

Docker Desktop a sa propre allocation disque. Augmentez-la dans Settings → Resources → Disk image size (passer à 64 Go par exemple).

Nettoyer le cache Docker avant si nécessaire :

docker system prune -a
# Supprime tous les conteneurs arrêtés, images non utilisées, etc.
kubectl get nodes : "The connection to the server was refused"

kubectl ne sait pas vers quel cluster pointer. Vérifier les contextes :

kubectl config get-contexts
kubectl config use-context k3d-td-k8s

Si le contexte k3d-td-k8s n'existe pas, le cluster n'a probablement pas été créé. Refaire la section 4.

Smoke test : page nginx pas accessible sur localhost:8080

1. Vérifier que le port mapping est bien actif :

docker ps --filter "name=k3d-td-k8s-serverlb"
# La colonne PORTS doit montrer 0.0.0.0:8080->80/tcp

2. Si pas de mapping → le cluster a été créé sans --port. Le détruire et le recréer (section 4).

3. Vérifier que le Service a bien un type LoadBalancer et le selector correct :

kubectl describe svc smoke-nginx
# Endpoints doit lister l'IP du pod, pas être vide
Docker pull : "toomanyrequests" ou "rate limit"

Docker Hub limite les pulls anonymes (100 par 6h depuis votre IP publique). Solution :

docker login
# Compte Docker Hub gratuit suffit, limite passe à 200 par 6h

Toujours bloqué ?

Contactez l'enseignant avant le jour du TD, pas le matin même. Précisez : OS + version, sortie de docker version, sortie de k3d cluster list, et le message d'erreur exact.
